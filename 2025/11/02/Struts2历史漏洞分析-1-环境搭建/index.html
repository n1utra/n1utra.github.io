Strust框架执行流程

Struts2的执行流程如下所示

Strust2框架执行中的元素

来自于：走进Struts2（一） — Struts2的执行流程及其工作原理 - mfmdaoyou - 博客园

key

可以看到图片中的key按照颜色区分了4种类型。

  • Servlet Filters：过滤器链，客户端的所有请求都要经过Filter链的处理。
  • Strust Core：Strust2的核心部分。
  • Interceptors：Strust2的拦截器。有一些是Strust2自带的默认拦截器，也可以自定义一些拦截器。
  • User Created：由业务人员创建，包括Struts.xml、Action、Template

FilterDispatcher

是整个Struts2的调度中心，也就是MVC中的C（控制中心），根据ActionMapper的结果来决定是否处理请求，如果ActionMapper指出该URL应该被Struts2处理，那么它将会执行Action处理，并停止过滤器链上还没有执行的过滤器。

ActionMapper

会判断这个请求是否应该被Struts2处理，如果需要Struts2处理，ActionMapper会返回一个对象来描述请求对应的ActionInvocation的信息。

Action

它会创建一个ActionInvocation实例，位于Action和xwork之间，使得我们在将来有机会引入更多的实现方式，比如通过WebService来实现等。

ConfigurationManager

是xwork配置的管理中心，可以把它看做struts.xml这个配置文件在内存中的对应。

struts.xml

是Stuts2的应用配置文件，负责诸如URL与Action之间映射关系的配置、以及执行后页面跳转的Result配置等。

ActionInvocation

真正调用并执行Action，它拥有一个Action实例和这个Action所依赖的拦截器实例。ActionInvocation会按照指定的顺序去执行这些拦截器、Action以及相应的Result。

Interceptor

是Struts2的基石，类似于JavaWeb的Filter，拦截器是一些无状态的类，拦截器可以自动拦截Action，它们给开发者提供了在Action运行之前或Result运行之后来执行一些功能代码的机会。

Action

由业务创建，负责处理请求、封装数据等。

Struts2执行流程

图中的request这里是已经被tomcat根据web.xml的规则送达到strust的状态，此时会先通过ActionContextCleanUp,这是防止tomcat线程复用时会产生线程干扰，这里不用管。然后会进入other filter，这个大多是业务编写的filter，比如权限校验等。然后会进入到FilterDispatcher，在FilterDispatcher中会调用ActionMapper寻找对应的Action。也就是FilterDispatcher和ActionMapper是调用关系，FilterDispatcher内部调用了ActionMapper。示例代码如下：

假如actionMapper匹配到对应的action，这里就会创建一个ActionProxy对象，但是创建时这个对象并不知道要执行哪个Action，所以需要通过ConfigurationManager读取Strust.xml(在启动时ConfigurationManager就已经读取了Strust.xml并进行了缓存)。当得到了需要执行的Action、相关的拦截器等信息之后，就会创建ActionInvocation对象。Action的整个调用过程都是由ActionInvocation负责。

然后在执行到对应的Action之前，会先经过拦截器1、2、3处理(这里是递归调用)，然后执行execute方法，得到结果Result，然后根据结果去strust.xml中匹配需要返回的页面，这个页面能够根据Strust2自带的标签库来访问数据，并生成最终页面。最后ActionInvocation对象再通过倒序运行拦截器即3、2、1(这里的内存栈是1 → 2 → 3 → Action → 3 → 2 → 1)，运行完成之后就得到的响应对象。最后按照与过滤器配置定义相反的顺序依次经过过滤器，最后返回响应结果。

初探OGNL表达式

OGNL表达式的学习全部来自于Drunkbaby师傅的文章，这里的内容也基本来自于Java Struts2 学习与环境搭建 | Drunkbaby’s Blog

OGNL表达式全称是Object-Graph Navigation Language，翻译一下叫对象图导航语言。通过表达式语法，能够存取对象的任意属性，调用对象的方法，遍历整个对象的结构图，实现字段类型转化等功能。

OGNL三要素

  • **表达式(Expression)**：表达式是整个OGNL的核心内容，所有的OGNL操作都是针对表达式解析后进行的。通过表达式来告诉OGNL到底要干什么。表达式其实就是一个带有语义的字符串，这个字符串规定操作的类型和内容。

  • Root对象：root对象可以理解为OGNL要操作的对象，当指定一个表达式时，需要指定这个表达式针对的是哪个对象。这个对象就是root对象。

  • 上下文环境：在OGNL内部，所有的操作比如对root对象进行赋值取值等操作，都是在一个特定的数据环境中运行。这个数据环境就是上下文环境(Context)。OGNL的上下文环境其实就是一个MAP结构，实现了java.utils.Map的接口，称之为OgnlContext。Root对象也会被添加到上下文环境中。

    区分上下文和根对象的主要目的是为了避免操作冲突和提高灵活性。

    • 避免命名冲突：上下文变量和根对象属性可能使用相同的名称
    • 数据隔离：将辅助数据（上下文）与主要操作对象（根对象）分离
    • 提高安全性：防止意外修改重要的上下文数据

投影与选择

OGNL 支持类似数据库当中的选择与投影功能。

投影：选出集合中相同属性组合成一个新的集合，语法为collection.{XXX}，这里的XXX就是集合中每个元素的公共属性。

选择：选择就是选择出集合中符合条件的元素组成新的集合，语法为collection.{Y XXX}，其中Y是选择操作符，XXX是选择用的逻辑表达式。

选择操作符有三种：

  • ?：选择满足条件的所有元素
  • ^：选择满足条件的第一个元素
  • $：选择满足条件的最后一个元素
