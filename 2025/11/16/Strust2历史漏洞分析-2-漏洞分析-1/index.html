S2-001

漏洞信息

漏洞版本： Struts 2.0.0 - Struts 2.0.8

漏洞环境：

漏洞描述： 此漏洞源于 Struts 2 框架中的一个标签处理功能：altSyntax。在开启时，支持对标签中的 OGNL 表达式进行解析并执行 [struts2 标签解析主要依赖于 xwork2, 可以理解为 xwork2 的漏洞]，攻击者通过使用 %{} 包裹恶意表达式的方式，将参数传递给应用程序，由于应用程序处理逻辑失误，导致了二次解析，造成了漏洞。

说简单一点，就是漏洞环境中登录失败时会将错误的用户名和密码重新渲染到页面中，在重新渲染的过程中识别到 %{} 就会通过 OGNL 执行被包裹的表达式

代码分析

之前写的几篇大多是从漏洞复现的角度去分析，这次打算按照自己的思路从漏洞发现的角度去分析。这也是在学习 drun1baby 师傅博客时看到的他的想法。

直接搜索 ognl.getValue，发现存在 4 处

全部打上断点，发送登录请求看看会不会停下，首先在 return 这个地方停下，那么看看这里面的值有没有我们传入的参数

放了几次，发现这里的 name 参数更多的是类似模板之类的，为了节约时间，把上面的断点取消，看看下面的 getValue，可以看到此时已经 getValue 接收到的值是 username，这是传入的参数名

我们直接跳过，看看下一个值是什么，可以看到这次获取的是 parameters.nameValue，parameters 属性在上下文中，由于这是通过上下文配置注册的特殊对象，所以访问时无需添加 #，可以看到，这个属性其实就是传入的用户名和密码参数。这个 compile 方法的租用是编译表达式为可执行对象，然后进行缓存管理。这里可以不用管。

这里这个方法在整个请求过程中总共调用了 4 次，每个参数名和参数值均分别调用了一次，并且参数名和参数值虽然都调用了这个方法，但是前面的路径却不同，所以我们现在看看参数名的调用过程。

参数名的调用过程

在调用栈中可以看到，在传递值为参数名时，前一个方法是 findValue，这里这个 overrides 的作用是表达式重写映射，但是这里为 null，不用管，也就是说参数在这里基本没有处理，继续向前面看调用。

findValue 是被 translateVariables 方法方法调用，也就是说如果可以给 findValue 直接传入表达式就能够实现执行 ognl。在这个方法打上断点，看看这里是怎么运行的，这里的参数流转是否可控。重新发送一个请求，将断点停在这个方法中，这里还会传入 index.jsp、login 这些值，但是运行到这些值的时候都是直接跳出循环的，并且我们无法控制，一直放行，直到看到 %{username}，此时是能够进入循环的。

可以看到在方法中传入的值最开始是 %{username}，然后在死循环中经过处理变成了 username，现在来看看这个循环是怎么流转的。

此时我们的参数是 %{username}，进入循环之后，首先通过 indexof 获得 %{ 的第一次出现的索引，如果没有则返回 -1，这里存在则是 0，然后获得整个参数的长度，后面定义了一个 x，在 start 的基础上进行了 +2，后面进入了一个循环，这个循环主要是找到占位符也就是 } 的位置，并且处理内部其他大括号的情况。

这个循环结束后就是一个 if 判断，在 if 中取出了占位符中间的值，也就是 username，然后通过 findValue 去查询 username 的值，这里查询出来的结果就是传入的参数值。然后简单拼接了一下，又进行循环，此时就是执行的传入的 1+1，但是这个值没有 %{} 等占位符，也就不会进入循环，更不会通过 findValue 去查询。

也就是说如果我们传入的参数也是通过 %{} 占位符包裹，那么就会将参数中的值传入 findValue 进行执行

重新构造一个请求值，用占位符将值包裹一下 %{1+1}，可以看到此时成功将值传递到了 findValue 中

执行一下，能够进行简单计算

试试弹出一个计算器，执行完 findValue 时，成功弹出计算器。

参数值的调用过程

查看参数值的传递过程，其实这里并不可控，这里传递的是 parameters.nameValue，这个值是在前面的拦截器部分处理的，在拦截器中存在一个 ParametersInterceptor，这个拦截器会将 HTTP 请求参数转换填充到 parameters Map 中，nameValue 是表单的 name 属性，对应的 value 就是用户输入的内容。

修复

官方修复代码，可以看到这里新增了一个 if (loopCount > maxLoopCount) 判断，也就是做了一个循环次数的判断。

总结

我们这里漏洞分析是通过查找调用逆向找到漏洞点的，这里从正向简单描述下

前面判断用户名和密码都错误，返回 error，根据 strust2.xml 配置文件，此时会跳转到 index.jsp。并且会原样返回用户输入的值。当返回用户输入的值并进行标签解析时，默认开启了 altSyntax，会调用 translateVariables 对标签中的 Ognl 表达式进行递归解析返回 ValueStack 值栈中同名属性的值

S2-003

漏洞信息

漏洞版本： Struts 2.0.0 - Struts 2.1.8.1

漏洞环境：

漏洞描述：

Struts2 会将 HTTP 的每个参数名解析为 ognl 语句执行由 ParametersInterceptor 调用 Ognl.setValue(...) 来将参数值写入 Action 对象, OGNL 表达式通过#来访问 Struts 的对象, Struts 框架中 ParametersInterceptor 通过过滤#字符防止安全问题, 通过 unicode 编码(u0023)或 8 进制(43)就可以绕过了这个安全限制从而可以操作服务端的上下文对象进行执行恶意的代码.

漏洞复现

payload：%28%27%20%5Cu0023context%5B%5C%27xwork.MethodAccessor.denyMethodExecution%5C%27%5D%5Cu003dfalse%27%29%28bla%29%28bla%29&%28%27@java.lang.Runtime@getRuntime%28%29.exec%28%5C%27calc%5C%27%29%27%29%28bla%29%28bla%29

需要注意网上文章中的部分 payload 可能需要额外的 url 编码

代码分析

这里为什么要传入()()形式：

如果传入 @java.lang.Runtime@getRuntime().exec('calc') 为例，由于漏洞点是 setValue，所以执行之后是给 exec(calc) 进行赋值，但是这个值并不是一个变量或者属性，不是一个合法的左值，所以无法完成赋值。

初次尝试

如果传入 ("@java.lang.Runtime@getRuntime().exec('calc')")(aaa)(bbb)，那么就会先去执行 @java.lang.Runtime@getRuntime().exec('calc')，获取到结果 o 之后再去调用 o.aaa，这里是先执行完毕之后，再去调用，所以这里的 aaa 就无关紧要了，因为已经执行完毕了。同时这个 payload 的位置也可以变一下，比如：(aaa)((payload)(bbb))，只需要保证 payload 能够在每一组 ()() 格式中占据左值即可。

在请求进入到 Action 之前，会经过一些拦截器 Interceptor 进行处理，其中有一个 ParametersInterceptor 拦截器，问题就是出在这里面。

在执行到 setParameters 方法时，首先将参数名与参数值放到 TreeMap 中，然后将这个 map 遍历，遍历时将参数等信息通过 Ognl 表达式填充到 stack 中，其中会执行到 stack.setValue(name, value);

进入这个 stack.setValue 方法，这里面其实执行的是 Ognl.setValue 方法

继续跟进，这里其实还是在调用 setValue，只是将参数中的表达式字符串通过 compile 转为了节点类型，可以看到，这里已经转为了 ASTEval 这个类型。

继续跟进 setValue 方法，这里就会去调用到 n.setValue 即 simpleNode.setValue

进入 simpleNode.setValue 这里首先从上下文中获取一个常量进行判断，默认为 false，所以直接进入 else 中的 evaluateSetValueBody 方法。而这个方法就是就是在根据节点类型，进入不同节点的 setValueBody 方法。

此时就进入了 ASTEval.setValueBody 方法中，这里的 children[0] 是 ("@java.lang.Runtime@getRuntime().exec(\'calc\')")(bla)，还是满足了 ()()，所以这里进入到 simpleNode.getValue 之后又会被分发到 ASTEval.getValue 中

但是这次进入跟之前的 setValue 的 evaluateGetValueBody 不同，可以看到这里额外多了一些判断，然后会进入到 getValueBody 中

那么此时就进入了 ASTEval.getValueBody，而这时的 children[0] 也变成了 "@java.lang.Runtime@getRuntime().exec(\'calc\')"，跟进 getValue 方法。前面都是 ognl 初始化的代码，一直执行到 node.getValue( context, source )

跟进一下，还是经过 simpleNode 进行分发，此时进入到 ASTChain.getValueBody 中，这里其实就是将 chain 拆分，直接进入了最后一个 if 判断中

经过分发，进入到 ASTStaticMethod.getValueBody，可以看到这里调用了一个 callStaticMethod 方法，跟进这个方法

这个方法其实是在进行反射调用进行方法执行，那么是否能够顺利执行呢？

运行到 callStaticMethod 方法，如果 e 不存在，那么就会调用父类的 callStaticMethod，这里首先从上下文中获取了一个常量 xwork.MethodAccessor.denyMethodExecution，这个值是 true，继续往下，这里得到的 e 也为 true，所以现在没办法进入 if，也就没办法执行。

但是这个值在上下文中，我们同样可以通过 OGNL 表达式进行修改，在 ParametersInterceptor#setParameters 是通过迭代器遍历每一个参数进行操作，所以可以将修改这个值的表达式作为第一个参数名，要执行命令的表达式作为第二个参数。

修改常量

('#context[\'xwork.MethodAccessor.denyMethodExecution\']=false')(bla)(bla) 这是修改常量的 payload，先尝试下能不能直接进行修改，调试时发现这里都不能进入到 stack.setValue 方法中，在这个方法前面还有一个对方法名进行了处理，具体在 acceptableName 方法中

可以看到，这里判断方法名中是否存在 =、,、#、: 等特殊字符以及满足 dojo\..* 的正则，如果存在则返回 false，但是我们要操作上下文就必须满足 # 和 =

在渗透测试中，通常遇到黑名单拦截关键字会通过 fuzz 字符编码去尝试绕过，那么想要后端能够解析，还需要有地方能够将字符进行还原，所以这里我们需要寻找能够还原字符的地方。这里已经知道使用 unicode 编码可以绕过，在 compile 方法中调用 parseExpression 时，new OgnlParser 的过程中会进行解析

具体的是会调用到 JavaCharStream.readChar 方法，可以看到这里在进行解码。这里的过程太麻烦不做过多描述

接下来我们直接进入后续步骤

由于这里的 payload 和前面差不多，都是先在 simpleNode 进行分发进入 ASTEval，然后又进入 1 次分发，此时就是执行我们的语句，所以我们直接进入最后一次分发，可以看到此时语句已经变成了 ASTAssign 子类

继续跟进 getValueBody，在这里的 node = (expr instanceof Node) ? (Node) expr : (Node) Ognl.parseExpression( expr.toString() ); 处，就将 payload 按照等号进行切分，分别赋值给了 children

然后跟进 node.getValue，此时就分发到了 ASTAssign 子类中，此时 children[0] 就是表达式，children[1] 就是 false，这里还是通过分发调用到子类的 setValueBody 方法，然后进行调用方法等，不再多说，最后是通过 map.put 将键名和值放到了上下文中

这里运行结束后，可以看到线程变量中上下文的 xwork.MethodAccessor.denyMethodExecution 已经变成了 false

最终构造

现在已经成功修改了上下文中的属性，应该没有什么能够阻止命令执行了，现在将两段 payload 拼接到一起，url 参数通过 & 拼接，发送请求，但是没有执行，这是为什么？

继续断点调试，发现在 ParametersInterceptor 中遍历时始终是命令执行最开始执行，而这个顺序是根据 TreeMap 决定的，TreeMap 默认按照键名的 ASCII 码升序排列，因为这里 \ 在 @ 后面，所以先执行的是命令执行。要改变就想办法改变两者的顺序，最简单的就是在修改属性前面增加一个空格，排名靠前并且不会影响语句。

修改之后，最开始执行的就始终是修改属性。

PS

在 2.0.11.2 版本之后添加空格的方式就不能用了，原因是在 Param 那个拦截其中遍历参数时的 boolean acceptableName = acceptableName(name) 这里的 acceptableName 方法变了，这里原本只是判断是否存在特殊字符，现在会在 isAccepted 方法中通过正则判断是否存在非控制字符和空白字符比如空格和换行，同时也会判断几个 #、= 等特殊字符。所以需要切换其他的 payload

其他的一些 payload 是在命令执行的前面添加了一个上下文临时变量比如 #ret=@java.lang.Runtime.....，这里的临时变量可以取任意值，是将右边的 payload 执行结束后的结果赋值到左边，只要不和原本的上下文变量相同导致覆盖即可。这样写有几个好处：

  1. 能够通过赋值的方式对 Ognl 进行链式调用。
  2. 需要处理命令执行的回显，命令执行之后还需要将结果赋值之后通过回显链返回。
  3. 能绕过一些关键字过滤。

修复

官方通过增加安全配置禁止静态方法调用(allowStaticMethodAcces)和类方法执行(MethodAccessor.den yMethodExecution)等来修补

S2-005

漏洞信息

漏洞版本： Struts 2.0.0 - Struts 2.1.8.1

漏洞环境：

漏洞描述： Struts 框架通过过滤#字符防止安全问题, 通过 unicode 编码(u0023)或 8 进制(43)就可以绕过了安全限制.S2-003 漏洞补丁中的安全配置(禁止静态方法 allowStaticMethodAcces、MethodAccessor.denyMethodExecution 调用和类方法执行等)被绕过再次导致了 S2-005 漏洞.利用 OGNL 先把沙盒关闭掉, xwork.MethodAccessor.denyMethodExecution 设置为 false, allowStaticMethodAccess 设置为 true 就可以绕过沙盒机制了

漏洞复现

payload：('\u0023_memberAccess[\'allowStaticMethodAccess\']\u003dtrue')(bla)(bla)&('\u0023context[\'xwork.MethodAccessor.denyMethodExecution\']\u003dfalse')(bla)(bla)&('\u0023ret\u003d@java.lang.Runtime@getRuntime().exec(\'calc\')')(bla)(bla)

代码分析

S2-005 是在 S2-003 的基础上做了一点修复，但是仍然可以被绕过，那么先尝试用 S2-003 的 payload 看看到哪一步的时候不能执行

一直调试直到走到 ASTAssign.getValueBody 这里前面 S2-003 的 payload 没走到这里是因为命令执行的语句不同，这里采用了加临时变量的方法绕过最开始的正则判断。可以看到左下角的那个常量已经处于 false 的状态，当前执行的语句是 #ret = @java.lang.Runtime@getRuntime().exec("calc")，也就是说只要 Object result = this._children[1].getValue(context, source); 这个语句执行成功就能够弹出计算器。

但是这里执行的时候抛出了异常

那跟进看看具体是怎么回事，跟进分发最后会进入到 ASTChain.getValueBody 中，这里就是 S2-003 中 payload 最终执行代码的地方，向下执行看看问题出现在哪里，继续向下执行会进入到 ASTStaticMethod 中，然后在 var10 = OgnlRuntime.callStaticMethod(context, this._className, this._methodName, args); 处抛出异常，继续跟进。

接下来进入 callStaticMethod，跟 S2-003 一样，这里会去判断上下文中 DENY_METHOD_EXECUTION 这个常量，但是这里已经设置了 false，所以能够直接进入 if 执行

继续向下跟进，直到进入 callAppropriateMethod 方法，我们在这里获取 method 之后，只要进入 if，就表示已经产生了错误，这里判断 method 是否为空或者是否可用，如果为空或者不可用就会产生报错。这里方法当然不为空，所以需要看看为什么不可用。

重新运行，进入 isMethodAccessible 方法，然后一直跟进到 isMethodAccessible 方法，可以看到，中间经过几次判断，最后就是 allow 变成了 false，然后最后返回了 false。让我们研究下这里的判断，首先是 Modifier.isStatic(modifiers) 判断是否是静态方法，如果是就进入 member instanceof Method && !getAllowStaticMethodAccess() 判断当前成员是否为方法成员，并且判断是否不允许访问静态方法。如果都满足就会设置 allow 为 false，其实问题的关键就在这里，后面的都不用看了。

getAllowStaticMethodAccess 是一个 getter 方法，那么就还有一个 setter 方法，这个值也是这个类本身的属性。

接下来的问题是应该怎么去修改这个值。这里涉及到 OGNL 引擎内部，在引擎内部定义了一套保留键和内部字段，这些值在调试时是看不到，他们不在底层的 map 中。这些值是在系统初始化的时候执行的

可以看到这里的 _memberAccess 就是 SecurityMemberAccess 这个类，也就是说我们可以直接通过 #_memberAccess 来操作 getAllowStaticMethodAccess 这里获取的值，只需要设置 allowStaticMethodAccess 属性即可。

按照之前构造 denyMethodExecution 的方式重新构造 payload：('\u0023_memberAccess[\'allowStaticMethodAccess\']\u003dtrue')(bla)(bla)&('\u0023context[\'xwork.MethodAccessor.denyMethodExecution\']\u003dfalse')(bla)(bla)&('\u0023ret\u003d@java.lang.Runtime@getRuntime().exec(\'calc\')')(bla)(bla)

可以看到此时就这里不会返回 false 了

出来之后，也绕过了抛出异常，进入了下一步

最终也是经过了一些处理，进入了 ognl.OgnlRuntime#invokeMethod 方法中，此时是获取到了 getRuntime()，继续向后运行执行 exec 方法弹出计算器

S2-007

漏洞信息

漏洞版本： Struts 2.0.0 - Struts 2.2.3

漏洞环境：

漏洞描述： 在 Struts2 中, 可以将 HTTP 请求数据注入到实际业务 Action 的属性中, 这些属性可以是任意类型的数据, 通过 HTTP 只能获取到 String 类型数据, Struts2 中默认有一个类型转换器, 可以完成大部分的自动转换操作, 可以通过 xml 文件, 来定义转换规则.比如 Action 类中有一个 integer 属性, 不需要执行任何操作, Struts 会自动将请求参数转换为 integer 属性.当配置了 Validation 时, 若类型转换出错, 后端默认会将用户提交的表单值通过字符串拼接, 然后执行一次 OGNL 表达式解析并返回, 从而可以构造特殊的恶意请求来执行命令.这种利用方式和 S2-001 的很相似, 不同的是利用点不同.

漏洞复现

payload: ' + (#_memberAccess["allowStaticMethodAccess"]=true,#foo=new java.lang.Boolean("false") ,#context["xwork.MethodAccessor.denyMethodExecution"]=#foo,@org.apache.commons.io.IOUtils@toString(@java.lang.Runtime@getRuntime().exec('whoami').getInputStream())) + '

代码分析

这个漏洞其实和 S2-001 很像，案例中 S2-001 是在回显时重新渲染用户输入的值时执行。而 S2-007 是在发生类型转换错误之后，重新渲染页面时生成字段错误信息时造成了命令执行。二者都需要重新渲染页面，但是漏洞触发场景不同，

在 ParametersInterceptor 获取参数之后，会将参数装载到值栈 ValueStack 中

具体发生在 callAppropriateMethod 这个类当中，这里其实就是 S2-005 的绕过关键点，但是本次是因为这里 getAppropriateMethod 结果为 null 造成必须进入 if 中。因在 getAppropriateMethod 这个方法中会找到最合适的方法，如果未找到方法就会尝试调用 getConvertedMethodAndArgs 进行类型转换，因为 setAge 方法接受的参数是 Integer，而我们传入的参数是字符串，在代码中无法转为 Integer，所以会直接抛出异常。

这里抛出异常之后最终会被记录到 ActionContext 的 conversionErrors 中，然后会继续执行拦截器栈，其中存在一个 ConversionErrorInterceptor 拦截器，这个拦截器就是专门用于处理类型转换错误的拦截器。可以看到这里面就存在输出错误后返回给前端显示的错误信息，在最后将错误的字段和值等信息放在了 fakie 中。

在放入的时候，会通过 getOverrideExpr 把这个值用单引号进行包裹

当这里拦截器执行完毕之后就会对 jsp 标签进行解析，剩下的流程就和 S2-001 比较类似了，也是在 findValue 时触发了命令执行

可以跟进看看之前的单引号是怎么影响 payload 的，可以看到，在将值装入 _children 的时候，已经按照 + 分割将 payload 分为三段，从而绕过了原本的单引号。需要注意的是，这次的 payload 位置在参数中，不能通过 & 拼接各个语句，而是需要一个整体语句进行传入。

修复

对转换出错的值进行了转义处理，从而无法闭合单引号。

官方定义中这里存在 4 种情况：

  • 类型转换 RCE，这里其实就是 S2-007
  • CookieInterceptor 中的 RCE，但是这个拦截器不在默认拦截器栈中，并且大多 web 容器都会对 Cookie 名称进行限制。基本阻拦了 ognl 的调用

S2-008 - Apache Struts 2 Wiki - Apache 软件基金会 — S2-008 - Apache Struts 2 Wiki - Apache Software Foundation

这里就说说 DebuggingInterceptor 这个版本

Struts2 框架存在一个 devmode 模式, 当 devmode 模式开启时, Struts2 对传入的参数没有严格限制, 导致多个地方可以执行恶意代码

漏洞复现

payload: ?debug=command&expression=%28%23_memberAccess%5B%22allowStaticMethodAccess%22%5D=true%2C%23context%5B%22xwork.MethodAccessor.denyMethodExecution%22%5D%2C@java.lang.Runtime@getRuntime%28%29.exec%28%27calc%27%29%29

代码分析

在 Struts2 的众多拦截器中，存在一个拦截器 DebuggingInterceptor，这个拦截器的主要作用就是用于调试。在这个拦截器的 intercept 方法中打上断点，直接访问系统。

此时停在断点处，初始化之后，通过一个三元运算符来获取调式模式的状态，此时环境中已经打开了调试模式，所以这里的值是 true，然后就会进入 if 判断。之后会从参数中获取一个常量 debug，根据 debug 的不同值进入不同的逻辑，这里判断了 4 种 MODE, 其中有一个是 COMMAND_MODE，这个值是 command，一看就是和命令相关的，所以我们这里重新发送请求，携带参数 debug=command

此时就会进入 else if (COMMAND_MODE.equals(type)) 这部分代码，看看这里的代码在做什么。

首先是创建 stack 值栈，然后对 session 进行了一点处理。然后是通过指定常量 EXPRESSION_PARAM 获取参数命令为 cmd，这里常量的值是 expression，也就是说这里还要传入一个参数。继续向下执行，进入 try 逻辑，可以看到这里将参数直接传入 findValue 进行执行。

现在可以试试直接将前面的 payload 作为 expression 的参数值一起传递，注意 payload 还需要绕过安全机制，以及作为参数值传入需要传入完整的语句。同时这个拦截器是在 ParametersInterceptor 拦截器之前运行的，所以还不需要考虑特殊符号绕过的问题。

最后也是成功执行。

漏洞信息

漏洞版本： Struts 2.0.0-Struts 2.3.1.1

漏洞环境： 直接使用 S2-003 或 S2-005

漏洞描述：

漏洞复现

payload：class.classLoader.jarPath=(#context["xwork.MethodAccessor.denyMethodExecution"]= new java.lang.Boolean(false), #_memberAccess["allowStaticMethodAccess"]=true, #a=@java.lang.Runtime@getRuntime().exec('calc'))(meh)&z[(class.classLoader.jarPath)('meh')]

代码分析

S2-009 其实就是 S2-005 的绕过，S2-005 只是增强了对参数名的过滤，但是没有处理参数值。所以可以把 payload 作为第一个参数的参数值传入，此时 struts2 会把参数放在 ValueStack 中，然后可以构造一个特殊的第二个参数，造成命令执行。

这里参数构造有个小坑，第一个参数名必须是 Ognl 可操作的属性名，ognl 可以在上下文中找到该对象，并且可以访问以及被认为可写。最简单来说就是自定义的 Action 类的属性，以及一些内置类。因为这里要放入第一个参数，但是 Ognl 不能凭空创造一个对象和属性，所以第一组参数存放是失败的，那么第二组参数就无法成功执行。

直接在 ParametersInterceptor 下断点，进入 setParameters，在前面的分析中能知道这里其实就是将参数放入 valuestack 的地方，此时已经接收到两组参数，第一组参数名是正常的属性，这里为了直观一点使用的 action 的属性作为参数名。所以不会产生什么问题，接下来重点关注遍历第二组参数时代码是如何流转的

这里直接跳到第二次遍历，可以看到已经把第一组参数放入了 newStack 中，一路跟进，这个过程中经过的 compile 会把 z[(message)('meh')] 这个参数名解析为 ASTChain，然后继续运行，经过分发之后就进入了 ASTChain 子类。

然后在 ASTChain 中会进行遍历，在此之前把这个参数装入 _children 的时候将 z 和 [(message)('meh')] 分开了

经过流转分发，最终进入了 ASTEval#getValueBody 方法中，可以看到这里已经根据 message 获取到了前面传入的 payload，后面的执行流程就和 S2-005 一样了，通过分发流转修改属性之后执行代码。

其中第二组参数写 z[()()] 是这是可以被解析的，同时可以在 ParametersInterceptor 将参数放入 TreeMap 时修改位置。还有一种写法是 top['foo'](0)，这种写法可以使用 top 来访问 Action 中的成员变量。都是为了让成员变量被调用时造成二次解析。
